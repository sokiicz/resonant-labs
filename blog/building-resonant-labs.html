Building Resonant Labs:
A Portfolio Site with No HTML Written

I had apps without a home. So I built one — a full portfolio site with dark space theme, animated background, tag filtering, cookie consent, comments, and newsletter. Every line written by AI. Here's exactly how it came together.

Before Resonant Labs existed, I had a problem that any solo creator eventually runs into: I was building things with no place to put them. KeepAwake was done. SpekPlatz was live. Music Mouse was working. But they lived scattered across GitHub repos, with no context, no presentation, no story connecting them.

I needed a home for them. And since I was already using Claude Code to build the apps themselves, building the website the same way felt like the obvious choice.

The concept first

Before writing — or describing — a single line, I spent time thinking about what this site actually needed to be. Not just technically, but in terms of feel. I wanted something that communicated the right things immediately: these are real apps, they're free, they're built differently, the person behind them is approachable.

That led to a clear brief: a deep dark space theme, a purple-to-cyan colour palette, app cards with screenshots and live links, a blog for documenting the journey, an about section that actually told the story, and — importantly — a way for people to contact me with ideas.

This is the part that doesn't get mentioned enough when people talk about AI-generated websites: the AI doesn't decide any of that. I thought through the whole concept, every section, every feature, every design decision. Claude Code's job was to implement what I described. The thinking, the product instincts, the vision — that's all human work. The implementation is the AI's.

"It's not that the AI builds the site on its own. It's that you build it by describing it — by conversation, not by writing syntax. Your thinking is the skill. The AI is the tool."

What actually got built

The full site ended up with more features than I originally planned, because once you're in flow the scope naturally expands. The final list:

  • Design system — CSS custom properties for every colour, spacing unit, shadow, and radius. One file to control the whole visual language.
  • Animated ambient orbs — Fixed background orbs that breathe gently, giving the dark theme a sense of depth and life.
  • App cards with tag filtering — Cards rendered from a simple data file. Filter by category. Each card links to a full detail page.
  • Blog with full post layout — Reading progress bar, table of contents sidebar, share buttons, article footer with tags.
  • Cookie consent (ConsentKit) — Built from scratch when I refused to pay for a third-party tool. Full GDPR, Google Consent Mode v2, four categories.
  • Comments (Cusdis) — Open-source comment system. No account needed to leave a comment. Only loads with consent.
  • Idea chatbot — A floating widget that collects app ideas from visitors and submits them by email. Built to encourage the community to suggest what to build next.
  • Newsletter signup — Buttondown integration for people who want to follow along.
  • Google Tag Manager + Analytics — Proper setup with consent mode so tracking respects user choices.

The development loop

Working with Claude Code on a site like this follows a clear rhythm. Describe a section, review what comes out, refine. Describe the next section. Find something that doesn't feel right, describe the fix. The iterations are fast — usually resolved in one or two exchanges.

The most important skill I developed wasn't technical. It was learning to describe visual and functional ideas precisely enough that the AI could implement them exactly. "A subtle glow behind the card on hover" is clearer than "make it look nice." "The orbs should breathe in place, scale up and down slowly, not drift across the screen" is better than "animate the background."

That precision comes from actually thinking through what you want before you ask. Which, again, is the human work. The AI handles the rest.

The moment it got recursive

At some point during the build, I needed a cookie consent widget. I looked at the available options, decided not to pay for any of them, and turned to Claude Code. The result was ConsentKit — a full open-source consent widget that now lives on every page of this site.

I built the tool for the site inside the same workflow I was using to build the site. Then I published that tool separately so anyone else can use it. The whole thing became recursive: the process produced not just the website, but a piece of open-source software that the website itself now runs on.

That moment crystallised something for me. The real output of this way of building isn't just the specific thing you set out to make. It's the byproducts — the smaller pieces, the tools, the solutions to problems you encounter along the way. Each one can be extracted, published, and given away.

What it cost

In money terms: a few dollars of API usage spread across many sessions. The domain. That's it. No designer. No developer. No agency. No monthly platform fee.

In time terms: a few focused weeks of work — most of it thinking, deciding, describing, reviewing, and iterating. The actual "typing" involved was writing descriptions and reading responses, not writing code.

For anyone considering building something similar: the barrier is much lower than you think. You don't need to understand CSS variables or JavaScript modules or build systems. You need to be able to articulate clearly what you want. If you can describe a product clearly to a colleague, you can build it with Claude Code.

What comes next

The site is a living thing. New apps get added as I build them. New posts go up when there's something worth writing about. The idea chatbot collects suggestions from visitors, and the most interesting ones will become actual apps.

I'm currently working on a time-tracking and invoicing tool — something I currently pay for across two different subscriptions. Same logic as ConsentKit: if you're paying for something simple, ask whether you could just build it instead. The answer is increasingly yes.

If you're reading this and you have an idea — an app you wish existed, a tool you're currently paying for that feels overpriced, a small thing that would make your work easier — send it. I'll build it and release it here, free for everyone.
