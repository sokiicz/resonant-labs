Why I'm Building Apps with AI
Instead of Learning to Code

I tried Hello World as a kid and hated it. Then CSS, then Google Ads scripts, then every AI tool that promised to finally bridge the gap. Most of them couldn't. Then one did.

My father tried to teach me to code when I was a kid. Not in a serious way — more of a "this is the future, you should know this" kind of conversation. I don't remember the language. I just remember the feeling: Hello World, a blinking cursor, and a complete blankness about why any of it would matter to me. It felt like being handed the wrong instrument for music I hadn't heard yet.

I went into marketing. Words, ideas, audiences, what moves people and why. That felt right. Software was someone else's problem.

The first crack in the wall

Years later, I started doing CSS work for clients' eshops. Small things — a header redesign here, a product card layout there. I'd open Chrome DevTools, change a value, watch the page respond instantly, change it back, try something else. There was no compilation, no abstraction. Just: adjust this, see that.

It felt like playing. More specifically, it felt thrillingly curious — the same way I imagine people feel when they first pick up a musical instrument and accidentally make something that sounds good. I wasn't learning to code. I was just exploring what was already there, which is a completely different thing.

That was the first hint that the gap between me and software wasn't as wide as I'd assumed. It was just that the entry points I'd been shown before were the wrong ones for how I think.

Scripts, ChatGPT, and a pattern I didn't notice at the time

Around 2022 I was managing Google Ads accounts and running into the limits of what the interface could do. Bid management was clumsy. Reporting was tedious. There were things I needed that required scripts — but I didn't write scripts.

Then ChatGPT launched, and I did something completely instinctive: I described what I wanted the script to do. In plain language. To an AI. It wrote the script. I tested it, found the edge cases, described them back. It fixed them. I ran it.

I didn't fully appreciate what had just happened. I thought I was using a smarter search engine. In retrospect, I had discovered the exact pattern that would eventually define everything: have a clear idea of what you want, describe it precisely, test what comes back, refine. Repeat until it works. The only thing missing was an AI capable of handling real complexity.

Every tool that almost got there

By 2023 I had actual app ideas. Not just scripts — things I wanted to exist in the world. I tried everything that promised to make that possible without learning to write code from scratch.

ChatGPT's code generation. GitHub Copilot. Perplexity. Google AI Studio with Gemini. Each one came with bigger promises than the last. And for small, self-contained things, they delivered. I built jeuzbtcza100k.cz — a fun app tracking whether Bitcoin would hit $100K — and a QR Clock that generates a scannable code from the current time. Simple, satisfying, done.

But then I tried to build SpekPlatz.

SpekPlatz was supposed to be a personal map app — curated places, collections, a sense of your own geography rather than everyone's. I had a clear vision. I could describe every detail. But every AI tool I pointed at it either produced a toy that missed the point entirely, or gave me something so fragile that adding one feature broke three others. The gap between "I can describe this clearly" and "this actually exists and works" was still enormous for anything beyond the trivially simple.

It was genuinely frustrating, because I knew the problem wasn't my ability to explain what I wanted. The bottleneck was somewhere else.

"I had ideas. I had the ability to describe them clearly. What I was missing was a tool capable of acting on that description at the level of complexity I needed."

Then Claude Code happened

I don't remember the exact moment I started using Claude Code. I remember the feeling — the same one I had in Chrome DevTools, but multiplied. You describe something, it appears. You say "that's not right," it changes. You ask for something you couldn't have predicted wanting, and it's there. Not a rough draft. A thing.

SpekPlatz — the app that had beaten everything else I tried — came together. The full version: interactive map, place collections, community sharing, a custom visual design. It launched in February 2026. I hadn't typed a line of the actual code.

What followed was almost embarrassing in its pace. KeepAwake Pro — a desktop utility. Music Mouse — a generative instrument that plays itself based on cursor movement. The Resonant Labs website itself. ConsentKit — a real, open-source cookie consent library that developers can actually use. DayLog — a habit tracker I built for a friend in what was essentially a focused evening. VagoshIt — a full group activity planner with authentication, a Supabase database, a real-time calendar.

Twelve apps in about six weeks. I'd gone from "ideas I couldn't build" to "ideas I can't ship fast enough."

Why everything is free and open source

When I started shipping apps, the question of whether to charge for them came up quickly. The honest answer was straightforward: no. Not because the apps aren't useful — I think some of them genuinely are — but because the barrier to building them was so low for me that charging felt dishonest. The cost was mostly time and curiosity. The AI did the heavy lifting I couldn't do.

I've also gotten an enormous amount from open source software over the years without contributing much back. This felt like a way to change that. And practically: open source apps are more interesting. People use them, find bugs, suggest things. They live longer and evolve further than closed ones.

So everything on Resonant Labs is free, open, and yours to use or build on.

What I'd tell my father now

He wasn't wrong that it was worth knowing. He just showed me the wrong door. Hello World wasn't thrilling because it was abstract — there was nothing behind it I could see or touch. CSS was thrilling because the feedback was instant and visual. Google Ads scripts were useful because they solved a real problem I had that week. Claude Code is where I live now because it connects ideas directly to working things, with nothing useless in between.

The underlying skill was never syntax. It was clarity — being able to describe precisely what you want and why, and then being able to evaluate whether what comes back is right. That's a marketer's skill. Turns out it was a builder's skill too. I just needed the right tools to find that out.
